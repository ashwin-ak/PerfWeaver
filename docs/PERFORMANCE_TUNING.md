# Performance Tuning Guide

This guide will help you optimize load tests generated by PerfWeaver for different scenarios.

## Adjusting Load Model Parameters

### Thread Count
Controls the number of virtual users (concurrent requests).

```yaml
loadModel:
  threadCount: 10  # Start conservative
```

**Guidance:**
- Start with 10 threads and gradually increase
- Monitor memory and CPU on load generator machine
- Production testing typically uses 100-10000 threads
- Use distributed load generation for higher counts

### Ramp-Up Time
Time needed to reach full thread count (gradual increase).

```yaml
loadModel:
  rampUpTime: 60  # Seconds
```

**Guidance:**
- Realistic user behavior: 30-120 seconds
- Stress testing: 0-10 seconds
- Soak testing: 5-10 minutes to full load

### Duration
Total test execution time.

```yaml
loadModel:
  duration: 300  # 5 minutes
```

**Guidance:**
- Baseline tests: 5-10 minutes
- Soak tests: 30+ minutes to find memory leaks
- Spike tests: 1-2 minutes (after reaching peak)

### Iterations
How many times each user repeats transactions.

```yaml
loadModel:
  iterations: 1  # Per thread
```

**Guidance:**
- 1 for realistic user simulation
- Higher for long-running tests to ensure data variation

## Think Time Tuning

PerfWeaver automatically calculates think times from HAR. Customize with:

```yaml
thinkTime:
  percentile: 75      # Use 75th instead of median
  minThinkTime: 500   # Minimum human-like delay
  maxThinkTime: 60000 # Cap outliers at 1 minute
```

**Think Time Percentiles:**
- **p50 (median)**: Typical user behavior
- **p75**: Slightly above average (conservative)
- **p90**: More realistic peak behavior
- **p95**: High-end users

## Resource Filtering

Optimize which requests are included in your load test:

```yaml
filters:
  ignoreExtensions:
    - png
    - jpg
    - css
    - js
  ignoreResourceTypes:
    - image
    - stylesheet
    - font
    - media
```

**Why filter?**
- Browsers cache static assets
- Focus load on meaningful API calls
- Reduce network overhead in tests
- Realistic server-side load patterns

## Correlation and Parameterization

Provide realistic data variation:

```yaml
parameterization:
  envVars:
    - BASE_URL
    - AUTH_TOKEN
    - USER_ID
  datasets:
    - users.csv
    - products.csv
```

**CSV Format (users.csv):**
```csv
username,password,email
user1,pass123,user1@example.com
user2,pass456,user2@example.com
```

## Load Profile Patterns

### Baseline Test
Normal steady-state load.

```yaml
loadModel:
  threadCount: 50
  rampUpTime: 60
  duration: 600      # 10 minutes
```

### Stress Test
Gradually increase load until failure.

```yaml
loadModel:
  threadCount: 1000  # High!
  rampUpTime: 0      # Immediate
  duration: 300
```

### Soak Test
Maintain load over extended period to find leaks.

```yaml
loadModel:
  threadCount: 100
  rampUpTime: 300    # Slow ramp
  duration: 3600     # 1 hour
  iterations: 10     # Multiple iterations
```

### Spike Test
Sudden increase in load.

Use stepped load profile in your adapter:
```
Phase 1: 50 users × 5 min
Phase 2: 500 users × 2 min (spike)
Phase 3: 50 users × 3 min (recover)
```

## Monitoring and Analysis

### Check Generated Reports

```bash
perfweaver analyze traffic.har
```

Look for:
- Average request duration (baseline)
- Think time distribution (realistic delays)
- Parallel requests (connection pressure)
- Transaction structure (user flows)

### Performance Metrics to Track

In your load test results:
- **Response Time**: P50, P95, P99 percentiles
- **Throughput**: Requests/second at target load
- **Error Rate**: % of failed requests
- **Server Resources**: CPU, Memory, Connections
- **Saturation Point**: Where performance degrades

## Troubleshooting Performance

### High Memory Usage
- Reduce thread count
- Lower connection timeout
- Enable connection pooling
- Check for memory leaks in code

### High CPU on Load Generator
- Distribute across multiple machines
- Reduce think time variation complexity
- Use simpler correlation patterns
- Lower logging verbosity

### Inconsistent Response Times
- Check for time-dependent behavior
- Parameterize dynamic values properly
- Ensure test data is realistic
- Consider network latency variation

### Test Doesn't Scale Linearly
- Check for bottlenecks (locks, single-threaded)
- Verify parameterization prevents conflicts
- Monitor server resource exhaustion
- Consider saturation point

## Best Practices

1. **Start Small**: Begin with realistic user counts (10-50)
2. **Iterate Gradually**: Increase load incrementally
3. **Separate Concerns**: Baseline → Stress → Soak
4. **Capture Realistic Data**: HAR files from real users
5. **Monitor Both Sides**: Client (load generator) and server
6. **Document Configuration**: Keep config in version control
7. **Replay vs Real**: Compare test results to production
8. **Environment Parity**: Test against production-like setup

## Tool-Specific Tuning

### JMeter
- GUI slower than CLI; use CLI for tests
- Increase heap size: `HEAP="-Xms512m -Xmx2g"`
- Disable GUI elements in non-GUI mode

### k6
- Built for cloud-native testing
- Adjust VU rampup for gradual tests
- Use check() for assertions

### Gatling
- Requires JVM; adjust `-Xmx` heap
- Compile before running for performance
- Use assertions for detailed reporting

### Locust
- Pure Python; lightweight
- Supports distributed swarm mode
- Good for API-focused testing

### Playwright
- Browser-based; heavier resource usage
- Best for API load mode (not browser automation)
- Useful for testing with Playwright Inspector
